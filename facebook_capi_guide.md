# دليل إعداد Facebook Conversions API 📊

هذا الملف يشرح باختصار شديد ما تم إضافته لربط الباك إند بـ (واجهة تحويلات فيسبوك) المعروفة باسم CAPI، لضمان وصول أحداث الشراء (Purchase) بدقة 100% حتى لو كان العميل يستخدم مانع إعلانات (Ad Blocker).

## 🛠️ الملفات التي تم تعديلها:

### 1. `backend/controllers/order.controllers.js`
تم إضافة كود مخصص داخل دالة `createOrder` يعمل كالتالي:
- بمجرد نجاح تسجيل الأوردر في قاعدة البيانات، يقوم الكود بتجهيز البيانات لفيسبوك.
- **تشفير البيانات (Hashing):** يقوم الكود بتشفير رقم هاتف العميل واسمه باستخدام `crypto.createHash('sha256')` (مكتبة أساسية في Node.js، لا تحتاج `npm install`). وهذا شرط أساسي من فيسبوك لضمان الخصوصية.
- **منع التكرار (Deduplication):** يتم إرسال `invoiceNumber` الخاص بالطلب تحت مسمى `event_id`.
- **الإرسال (Fetch):** يتم استخدام دالة `fetch` (المدمجة في Node.js الحديث) لإرسال طلب POST لفيسبوك يحتوي على قيمة الطلب والمنتجات والبيانات المشفرة.

### 2. `backend/.env`
تمت إضافة المتغيرات البيئية التالية لضمان اتصال السيرفر بفيسبوك:
- `FACEBOOK_PIXEL_ID`
- `FACEBOOK_ACCESS_TOKEN`

### 3. `bm-store-arabic/src/utils/facebookPixel.js` (في الفرونت إند)
تم تعديل دالة `trackPurchase` السطر 200 لتصبح هكذا:
\`\`\`javascript
window.fbq("track", "Purchase", contentData, { eventID: invoiceNumber });
\`\`\`
هذا التعديل الصغير يجعل المتصفح يرسل نفس رقم الفاتورة `invoiceNumber` مع الحدث.

## 💡 كيف يعمل النظام الآن؟ (آلية منع التكرار)
عندما يقوم عميل بالشراء:
1. يرسل **المتصفح (الفرونت إند)** حدث `Purchase` ويحمل `eventID = INV-2023...01`.
2. يرسل **السيرفر (الباك إند)** نفس الحدث ويحمل `event_id = INV-2023...01`.
3. يرى فيسبوك الحدثين بنفس رقم الـ ID في وقت واحد، فيقوم بدمجهم معاً (Deduplication) ولا يحسبهم بيعتين.
إذا فشل المتصفح في الإرسال (بسبب Ad Blocker أو سوء إنترنت)، يعتمد فيسبوك على حدث السيرفر.

**تم التنفيذ بأفضل ممارسة (Clean Code) بدون الاعتماد على مكتبات خارجية ثقيلة (No Axios, No Facebook SDK).**
